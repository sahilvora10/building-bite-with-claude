# v0.3 Bugs Found and Fixes

**Date:** 2026-04-05 / 2026-04-06

## Bug 1: Rings not nested (Apple Fitness style)

**Symptom:** Rings were displayed side by side as individual circles, not nested concentrically like Apple Fitness.

**Fix:** Rewrote `ProgressRing.tsx` as `NestedRings` — renders concentric circles with decreasing radius per ring. Outer = calories, inner = fat. Added legend below with colored dots.

**Lesson:** Be specific with visual references in the spec. "Apple-style rings" could mean many things — nested concentric was the intent.

---

## Bug 2: Ring overflow past 100% — multiple failed approaches

**Symptom:** When a ring exceeded 100%, no visual distinction from a full ring.

**Attempts:**
1. Semi-transparent overlay at 60% opacity — too subtle
2. White-tinted arc with dark shadow separator — looked weird on device (pastel disc appearance, screenshot confirmed)
3. **Final fix:** Removed overlay entirely. Rings cap at 100% visually. Legend shows percentage text (e.g., "203%") when over goal. Simple and clear.

**Lesson:** Overengineering visual effects for edge cases can make things worse. Sometimes the simplest approach (text indicator) beats a clever visual one. Also: always test on device — SVG rendering on Android can look very different from expectations.

---

## Bug 3: Ring background too opaque

**Symptom:** Background rings used hex color + `25` suffix for opacity. On Android this rendered as solid pastel fills, making rings look like filled discs instead of thin arcs.

**Fix:** Changed to `opacity={0.15}` prop on the Circle element instead of encoding opacity in the color string.

**Lesson:** SVG color opacity via hex suffix (`#FF3B3025`) doesn't always render correctly on Android. Use the `opacity` prop directly.

---

## Bug 4: Meal cards not visible (layout)

**Symptom:** Rings dashboard took up full screen height, pushing meal cards off-screen with no way to scroll to them.

**Root cause:** Rings card was outside the ScrollView. Input section + rings card filled the viewport, leaving no room for meals.

**Fix:** Restructured layout — input section stays fixed at top, everything below (rings + meal cards) is inside a single ScrollView. Removed duplicate rings section that was accidentally left in.

**Lesson:** When adding large UI components, test scroll behavior. Fixed + scrollable content needs explicit layout separation.

---

## Bug 5: Meal card flickering (persistent across multiple fix attempts)

**Symptom:** All existing meal cards flicker when a new meal is logged.

**Attempts (chronological):**
1. `memo` on MealCard — no effect
2. `useCallback` on renderMeal — no effect
3. FlatList with `getItemLayout`, `removeClippedSubviews` — no effect
4. Replace FlatList with ScrollView — no effect
5. Extract `MealList` as memo component with custom comparator — no effect
6. Remove Android `elevation` shadows + batch state into single object — no effect
7. Current: removed MealList wrapper, cards rendered directly in ScrollView via .map()

**Status:** Still investigating. May be inherent to React Native re-rendering the ScrollView content when parent state changes. Could try `useReducer` or moving meal list to a completely separate component tree.

**Lesson:** Android flickering on re-render is a deep React Native issue. Multiple standard fixes (memo, elevation removal, state batching) failed. This may require architectural changes or accepting a small visual trade-off for v0.3.
